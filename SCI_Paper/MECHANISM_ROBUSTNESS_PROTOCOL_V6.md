# Prespecified mechanism-robustness experiment (v6)

**Protocol identifier:** `saos_mechanism_robustness_v6_20260722`  
**Frozen before result inspection:** 2026-07-22  
**Purpose:** distinguish state-representation, behavior-cloning, and rollout-update explanations for the v5 SA-AOS result without changing the FJSP-AGV backbone.

## 1. Scope and status

This is a stratified mechanism study, not a replacement for the 50-instance primary experiment and not a broad PPO hyperparameter search. All configurations use the v5 encoding, decoder, initialization mixture, ten operators, reward, archive, NSGA-III selection, population size 100, and deterministic instance-extension seeds. Every configuration is rerun under the same code revision and server environment; frozen v5 results are used only as an external replication check.

## 2. Instances, seeds, and budgets

The instances were selected before inspecting v6 outcomes to span both benchmark families and a broad size order:

- Brandimarte: `Mk01`, `Mk03`, `Mk05`, `Mk08`, `Mk10`;
- Hurink edata: `la01`, `la10`, `la20`, `la30`, `la40`.

Seeds are 42–51. Seeds are repeated runs nested within each instance; the ten instances, not the 100 run rows, are the inferential units.

Two budget blocks are run:

### Block M100: state and behavior-cloning mechanism, 100 generations

1. `UCBOnly`;
2. `Original24BC_R16`: original 24-dimensional state under the corrected independent-RNG implementation, BC, rollout 16;
3. `Original24NoBC_R16`: the same independent-RNG baseline, no BC, rollout 16;
4. `BasePaddedBC_R16`: the original 24 features followed by 54 constant zeros, BC, rollout 16;
5. `BasePaddedNoBC_R16`: the same capacity-matched padded state, no BC, rollout 16;
6. `EnhancedBC_R16`: enhanced state, BC, rollout 16;
7. `EnhancedNoBC_R16`: enhanced state, no BC, rollout 16.

The capacity-matched padded and enhanced configurations form the 2 × 2 state-information-by-BC mechanism design. The original 24-dimensional pair checks implementation replication but is not used alone to claim a pure state-information effect because its first layer contains fewer parameters.

### Block M200: rollout configuration under the enhanced state, 200 generations

1. `UCBOnly`;
2. `EnhancedBC_R8`;
3. `EnhancedBC_R16`;
4. `EnhancedBC_R32`.

The 200-generation block is used because, after a typical late handover, all three rollout settings can produce action-effective updates. Rollout length is interpreted as a compound update-frequency configuration, not as an isolated minibatch-size effect.

The complete design contains 1,100 runs: 700 in M100 and 400 in M200.

## 3. Enhanced state

The original 24 features are retained. Fifty exact-context features are added in five ten-action vectors:

1. recent-window selection counts divided by the window length;
2. cumulative counts divided by the generation budget;
3. a finite monotone transform of the ten UCB scores;
4. the ten forced-coverage indicators used by the UCB rule;
5. the ten zero-window-count indicators.

Four scalar summaries are appended: forced-coverage active, transition coverage ready, the fraction of operators present in the recent window, and the top-two UCB-score margin after a monotone finite transform. Total dimension: 78. No stage indicator is supplied: this avoids a feature that is always zero in the UCB demonstrations but always one during PPO deployment. For the capacity-matched base, the final 54 inputs are fixed at zero, so its actor and critic have the same parameter count as the enhanced controller. The UCB controller itself is unchanged. After handover, the same sliding statistics continue to be updated with the actions and rewards actually generated by PPO, but UCB no longer selects actions; these features are therefore current counterfactual UCB-context summaries rather than a frozen handover snapshot. Feature variance and saturation are audited separately before interpretation.

## 4. Training diagnostics retained

Each run stores:

- BC pre-training and final loss, accuracy, entropy, per-epoch curves, pre/post-BC KL divergence, and confusion matrix;
- demonstration label counts, normalized entropy, UCB margins, and standardized nearest-neighbor label disagreement;
- transition generation and reason;
- action-effective PPO updates completed before a later action, terminal full-buffer updates, and terminal residual updates;
- rollout sizes, optimizer steps, policy loss, value loss, approximate KL, clip fraction, entropy, gradient norm, pre-update explained variance, advantage scale, return scale, and reward scale;
- learning-module time, total CPU time, operator counts, and final fronts;
- enhanced-feature mean, standard deviation, minimum, maximum, and saturation rate during the PPO phase.

Terminal updates are not counted as action-effective because they cannot change the completed schedule.

Network initialization, BC shuffling, stochastic policy actions, and PPO minibatch permutations use four independently derived random streams. For a given instance and seed, all mechanism configurations receive the same stream seeds; BC shuffling therefore cannot advance the policy-action stream, and the 24- versus 78-dimensional initialization cannot advance either training or action-sampling streams. The derived seeds are stored in every run row and front payload.

## 5. Performance indicators and fixed normalization

The primary outcome is three-objective hypervolume recomputed from saved final fronts. For M100, ideal, nadir, and reference values are imported unchanged from the frozen v5 E1 `normalization.json`; for M200 they are imported from the frozen v5 E3 200-generation block. New methods do not change these bounds.

Before computing HV, the analysis must report whether any normalized objective is below 0, above 1, or not dominated by the fixed reference point `(1.1, 1.1, 1.1)`. No silent clipping is permitted. If the reference point fails to dominate a run, the affected HV comparison is marked invalid and the protocol is revised transparently before inferential interpretation.

The prespecified scale-robust secondary indicator is IGD+ against a frozen nondominated reference set constructed only from the corresponding v5 fronts under the same instance and budget. New v6 methods do not alter that reference set. HV is the only confirmatory endpoint. IGD+ is a supportive sensitivity analysis and is not used independently to declare a positive result; its within-endpoint Holm values are reported only for transparency, without cross-endpoint confirmatory interpretation. Best makespan, energy index, workload balance, and nondominated-set size are descriptive auxiliary outcomes and do not replace the prespecified HV comparisons.

## 6. Statistical plan

Within each instance and configuration, the ten seeds are collapsed by their median. Exact/two-sided paired Wilcoxon tests use the ten matched instances. Report the median paired difference, 95% instance bootstrap interval, wins/ties/losses, and paired rank-biserial effect.

Prespecified comparison families are:

### M100 state family (Holm within two tests)

- `EnhancedBC_R16 − BasePaddedBC_R16`;
- `EnhancedNoBC_R16 − BasePaddedNoBC_R16`.

### M100 BC family (Holm within two tests)

- `BasePaddedBC_R16 − BasePaddedNoBC_R16`;
- `EnhancedBC_R16 − EnhancedNoBC_R16`.

### M100 interaction family

- difference in differences: `(EnhancedBC − EnhancedNoBC) − (BasePaddedBC − BasePaddedNoBC)`.

### M200 rollout family (Holm within two tests)

- `EnhancedBC_R8 − EnhancedBC_R16`;
- `EnhancedBC_R32 − EnhancedBC_R16`.

### UCB boundary family (Holm within two tests)

- best prespecified enhanced candidate `EnhancedBC_R16 − UCBOnly` at M100;
- `EnhancedBC_R16 − UCBOnly` at M200.

The UCB comparisons are not selected after observing the data. No all-pairs testing is planned. Family-specific and instance-family summaries are reported as exploratory context.

## 7. Integrity gates

Before analysis, require:

- exactly 1,100 unique run keys and all expected front files;
- finite objective values and nonempty fronts;
- no duplicate keys;
- exact seed/configuration/instance grids;
- one protocol identifier only;
- fixed-reference dominance audit;
- recorded environment manifest and code hash;
- a semantic hash manifest for the ten benchmark input files (UTF-8 whitespace-token stream, so platform line-ending conversion does not alter identity);
- exactly 40 recorded worker processes for all formal runs;
- an atomic front write, per-front SHA-256 in the CSV, and a combined front-manifest digest in the completion marker;
- no mixing with the v5 output directory.

Any failed task is recorded; partial results are not used for inference.
